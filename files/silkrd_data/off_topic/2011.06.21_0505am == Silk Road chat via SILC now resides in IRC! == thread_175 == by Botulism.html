MAC SILC TUTORIAL

1. install macports from www.macports.org
2. sudo port install silc-client
3. sudo port install socat
4. READ THIS AND GENERATE A NEW KEY:

Q: How can I generate a new server key pair?
A: You can generate a new key pair using the silcd command with the -C option. When SILC Server is installed a key pair is generated automatically for you. However, it is suggested that you check the information found in that key and generate a new key pair if the information is incorrect. You can check the information of your public key by giving command: silc -S file.pub.
 
If you want to generate a new key pair then you can give for example command: silcd -C . --identifier="UN=silc-oper, HN=silc.silcnet.org, RN=SILC Router Admin, E=silc-oper@silcnet.org, O=SILC Project, C=SK". This will create the key pair to current directory, with the specified identifier. Please, give the --help option to the silcd to see usage help for the -C and --identifier options.

5. put the following in a script and run it:

socat TCP4-LISTEN:4242,fork SOCKS4A:localhost:fxb4654tpptq255w.onion:706,socksport=9050 &

6. fire up "silc" from command line then type /server localhost 4242
7. /j silkroad
8. party!

Here are some fairly simple instructions on using SILC on OSX Lion, based on a previous post about socat. socat seems broken via MacPorts at the moment, but it's easy to compile manually. Note that if using the Tor browser bundle it will assign a random Tor port every time you start it up, so you may want to install plain Vidalia instead.

Code: [Select]
#!/bin/bash
#
# Install Xcode & MacPorts
#
# sudo port install silc-client
#
# Manually build socat-1.7.2.0:
#   wget http://www.dest-unreach.org/socat/download/socat-1.7.2.0.tar.gz
#   tar zxvf socat-1.7.2.0.tar.gz
#   cd socat-1.7.2.0
#   ./configure
#   make
#   sudo make install
#
# WARNING: double-check your ~/.silc/silc.conf and scrub all traces of your name/nick/hostname/etc
# /whois yourself once connected to make sure it's using the right values
#

socat TCP4-LISTEN:4242,fork SOCKS4A:localhost:fxb4654tpptq255w.onion:706,socksport=9050 &
/opt/local/bin/silc -c localhost -p 4242

Someone should come up with a more elegant script, since socat will stay running in this case if you quit out of silc. But this gets the job done.

Here are some Mac instructions I posted a while ago in this thread, sorry but it takes a bit of work:

Here are some fairly simple instructions on using SILC on OSX Lion, based on a previous post about socat. socat seems broken via MacPorts at the moment, but it's easy to compile manually. #!/bin/bash
#
# Install Xcode & MacPorts
#
# sudo port install silc-client
#
# Manually build socat-1.7.2.0:
#   wget http://www.dest-unreach.org/socat/download/socat-1.7.2.0.tar.gz
#   tar zxvf socat-1.7.2.0.tar.gz
#   cd socat-1.7.2.0
#   ./configure
#   make
#   sudo make install
#
# WARNING: double-check your ~/.silc/silc.conf and scrub all traces of your name/nick/hostname/etc
# /whois yourself once connected to make sure it's using the right values
#

killall socat
sleep 1
socat TCP4-LISTEN:4242,fork SOCKS4A:localhost:fxb4654tpptq255w.onion:706,socksport=9050 &
/opt/local/bin/silc -c localhost -p 4242

Here are some Mac instructions I posted a while ago in this thread, sorry but it takes a bit of work:

Here are some fairly simple instructions on using SILC on OSX Lion, based on a previous post about socat. socat seems broken via MacPorts at the moment, but it's easy to compile manually. Note that if using the Tor browser bundle it will assign a random Tor port every time you start it up, so you may want to install plain Vidalia instead.

Code: [Select]
#!/bin/bash
#
# Install Xcode & MacPorts
#
# sudo port install silc-client
#
# Manually build socat-1.7.2.0:
#   wget http://www.dest-unreach.org/socat/download/socat-1.7.2.0.tar.gz
#   tar zxvf socat-1.7.2.0.tar.gz
#   cd socat-1.7.2.0
#   ./configure
#   make
#   sudo make install
#
# WARNING: double-check your ~/.silc/silc.conf and scrub all traces of your name/nick/hostname/etc
# /whois yourself once connected to make sure it's using the right values
#

killall socat
sleep 1
socat TCP4-LISTEN:4242,fork SOCKS4A:localhost:fxb4654tpptq255w.onion:706,socksport=9050 &
/opt/local/bin/silc -c localhost -p 4242

Sidenote: Requested by b4b33 to include this in my tutorial, how to block whiteboard requests! Whiteboard is a pain and is really buggy, popping up excessive windows and making it impossible to type. To disable, simply go to accounts>silc account>edit account>advanced>block messages on whiteboard.

If anyone else has some pointers let me know. Good luck!

This is based on the Linux v2.10 or so Pidgin, FYI: go to your accounts list.  Click the "add" button.  Select "irc" out of the list.  In the "Basic" tab: set your local alias to your desired username.  Set your Username to your desired username.  Set the Server to be "silcroadg3c3mtu6.onion".

Now go to the advanced tab.  Set "Port" to be 6667.  Encodings to UTF-8.  Uncheck Auto-detect incoming UTF-8.  Set username to your desired username.  Set Real name to the same desired username.  Uncheck "Use SSL."

Now on the proxy tab: set Proxy type to "Tor/privacy (SOCKS5)".  Set Host to 127.0.0.1.  Set port to 9050.  Leave Username and Password blank on the proxy tab.

Now sign on.


P.S. - if this still doesn't work, execute this command at a terminal window (run "xterm" or "gnome-terminal" and paste it or type it in), then restart the Tor browser and try signing in again and it should work fine:
Code: [Select]
echo "SocksPort 9050" >> /etc/tor/torrc; echo "SocksListenAddress 127.0.0.1" >> /etc/tor/torrc

Disclaimer: I don't use pidgin.  But it worked the one time I signed in with it set this way, so... should do fine.

Post by: SelfSovereignty on February 21, 2013, 12:35 am
So that means Time Till Live expired.  When a packet gets sent to the network, every time a computer picks it up and then re-transmits it because it isn't the recipient (that's how the internet works, believe it or not -- packets just get passed back and forth all day long until they hit the recipient), 1 gets subtracted from the packet's Time Till Live number.  If that hits 0, no computers re-send it, they just drop it where it is and notify the sender.

So basically, your connection got passed around for so long, it reached the limit your computer set for it and died.  Keep trying, if that's the only problem you're having then you'll get it as soon as you have a Tor circuit to the server.

Post by: astor on July 03, 2013, 08:12 pm
When you send a packet, it gets stamped with the number 127, and each router that it passes through decreases it by one. When it reaches zero, the Time To Live has expired and the packet is no longer forwarded anywhere. This feature was implemented because people thought internet packets might get stuck in loops, so if they don't reach their destination in 128 hops, they are killed.

I've seen TTL expired errors when there's a routing issue, like when two routers keep sending the packets back and before between each other until the TTL expires. Not sure what would be causing it for you.
